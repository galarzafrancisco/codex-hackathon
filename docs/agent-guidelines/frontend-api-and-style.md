# Frontend API and Style Guidelines for Agents

Use this as the default contract for UI work. The goal is consistent frontend code that follows the generated backend contract and the shared design system.

## Backend Access

- UI code consumes backend APIs only through the autogenerated client package.
- Do not add direct `fetch`, `axios`, raw `XMLHttpRequest`, or hand-written HTTP wrappers in feature code.
- Do not hand-write frontend copies of backend DTOs. Import generated request, response, and enum types from the generated client.
- Keep generated client usage inside feature data-access modules, hooks, or dedicated feature services.
- UI primitives must never import backend clients or feature data modules.
- Components should receive data and callbacks through props or feature hooks; they should not know transport details.
- Auth is cookie-based unless the backend contract says otherwise. Do not store or manually attach auth tokens in the UI.
- WebSocket events are incremental updates only. Load authoritative state through REST using the generated client first.

## Error Handling

- Parse backend Problem Details responses and key UI behavior on stable `code` values.
- Do not key UI logic on `title`, `detail`, exception class names, or raw message strings.
- Include `requestId` in support/debug surfaces when available.
- Feature screens must handle loading, empty, expected error, and unexpected error states.

## Styling System

- `tokens.css` is the source of truth for colors, spacing, radius, typography, shadows, and focus rings.
- Components use semantic CSS variables such as `var(--surface)`, `var(--text-muted)`, `var(--space-4)`, and `var(--r-2)`.
- Do not hardcode raw colors, one-off spacing, one-off radius values, or theme-specific styles inside components.
- Do not invent a new visual style per component. Reuse existing primitives, variants, tokens, and layout patterns first.
- If a reusable visual pattern is missing, add or extend a primitive or token intentionally instead of copying CSS into a feature component.
- Themes override tokens, not component CSS.
- Pages should mostly compose feature components and primitives; page-specific CSS should be minimal.
- Feature components may have CSS for domain-specific layout, but they should still use tokens and primitives.

## Component Boundaries

- `ui/` owns primitives and visual foundations.
- `features/` owns domain-specific components, state, and generated-client usage.
- `app/` owns shells, router, providers, and cross-feature composition.
- `ui/` must not depend on `features/`.
- `features/` may depend on `ui/`.
- Shells own app-level navigation. Features own in-app navigation.
- Do not fork route trees or duplicate page logic for mobile and desktop.
- Prefer CSS-first responsiveness. Use JavaScript viewport checks only for real behavior differences.

## Review Checklist

- No manual HTTP calls.
- No hand-written DTO mirrors.
- Generated client usage is contained in feature data-access code.
- No raw colors, magic spacing, or ad-hoc component styling.
- New styling either uses existing tokens/primitives or adds a shared token/primitive with clear intent.
- Loading, empty, error, and success states are present for data-driven views.
- Mobile and desktop share route content and differ through shell/layout behavior.
